# 프로젝트 리팩토링 및 고도화 보고서 (Refactoring & Optimization Report)

본 보고서는 프로젝트의 코드 품질 개선, 속도 최적화, 데이터 중복성 제거 및 연속 대화 메모리 기능 고도화 작업의 상세 내역과 기술적 근거를 정리한 문서입니다.

## 🛠️ 1. 작업 내용 (Implemented Modifications)

### ① 공통 설정 및 유틸리티 모듈 신설
*   **[config.py](file:///c:/STUDY/project/config.py) 생성**: 기존 소스 코드 곳곳에 중복으로 흩어져 하드코딩되어 있던 ChromaDB의 로컬 저장 디렉토리 경로(`./chroma_db`), 법령·체크리스트·FAQ 컬렉션 이름, 그리고 로컬 BM25 캐시 파일명들을 한곳에 선언하여 중앙 관리하도록 일원화하였습니다.
*   **[utils.py](file:///c:/STUDY/project/utils.py) 생성**: 각 파일별로 개별 구현되어 있던 Kiwi 형태소 분석기 초기화 및 한국어 전용 명사/용언 토크나이저 함수(`kiwi_tokenize`), ChromaDB 전체 문서 추출 함수(`get_all_documents_from_chroma`)를 공통 유틸리티 모듈로 이관 및 병합하여 불필요한 코드 중복을 제거하였습니다.

### ② 데이터 파이프라인 중복 적재 방지 및 멱등성 확보
*   ** Chunks Ingest 초기화 적용 (`ingest.py`, `ingest_law.py`, `FAQ.py`)**: 데이터 수집기를 여러 번 실행해도 데이터가 중복해서 누적되지 않도록, 데이터 저장 전 기존 ChromaDB 컬렉션을 조회하여 존재하던 이전 Document ID 목록을 일괄 삭제(`temp_store.delete(ids=...)`)하는 초기화 로직을 주입하였습니다.

### ③ BM25 인덱스 직렬화 캐시 적용 (Serialization)
*   **인제스트 단계 사전 빌드**: 데이터를 최초 적재하는 시점에 해당 문서 청크 리스트를 기반으로 `BM25Retriever` 객체를 미리 색인(Inverted Index)하고, 파이썬 객체 직렬화 라이브러리인 `pickle`을 이용해 지정된 `.pkl` 로컬 디렉토리 파일로 저장하도록 구현하였습니다.
*   **쿼리 단계 고속 로드**: `query.py` 구동 시 매번 ChromaDB 전체를 역인덱싱하던 오버헤드를 제거하고, 로컬 디스크에 사전 생성된 `.pkl` 파일이 있을 경우 `pickle.load`를 활용해 즉시 메모리에 올려서 하이브리드 리트리버를 초기화하도록 최적화하였습니다.

### ④ 5턴 연속 대화 기억 장치(Conversation Memory) 구축
*   **대화 이력 슬라이딩 윈도우 관리**: `query.py` 내의 `run_interactive_loop` 함수가 구동되는 동안 유저의 질문과 AI의 실시간 스트리밍 답변 이력을 `chat_history` 메모리 버퍼 리스트에 실시간 적재합니다. 메모리 누적으로 인한 LLM 토큰 수 제한(RESOURCE_EXHAUSTED)을 방지하기 위해 최대 5턴(질문/답변 쌍 10개)까지만 유지하는 슬라이딩 윈도우(Sliding Window) 제어를 수행합니다.
*   **검색 쿼리 재구성(Question Condensation)**: 사용자가 대화 맥락이 이어지는 생략형 질문(예: "그럼 바닥재는?")을 던졌을 때, `CONDENSE_QUESTION_TEMPLATE`를 기반으로 이전 대화 기록을 분석하여 의미가 생략되지 않은 온전한 검색 쿼리(예: "바닥재 인테리어 공정 시 시공 순서 및 확인 사항")로 재설정한 뒤 ChromaDB 검색에 활용하도록 설계하였습니다.

## 📈 2. 리팩토링 이유 및 기술적 근거 (Rationale)

*   **RAG 검색 정확도 저하 방지**:  Chunks Ingest가 중복해서 실행되어 DB 내에 동일한 텍스트를 가진 문서 데이터가 중복 보관될 경우, 하이브리드 리트리버가 RAG 답변용 context로 동일한 문서들만 수집하게 되어 한정된 LLM 입력 단락 안에 다양한 지식을 담지 못하는 현상이 발생하므로 DB 멱등성을 사전에 확보하였습니다.
*   **시스템 초기 로딩 레이턴시(Latency) 단축**: `BM25Retriever`는 기본적으로 전역 메모리에 역색인 표를 상주시키는 구조입니다. 매번 프로그램을 켤 때마다 DB의 모든 로컬 노드를 스캔해 색인을 빌드하는 것은 데이터가 증대됨에 따라 부하가 기하급수적으로 늘어나므로, 디스크에 사전 연산된 바이너리를 캐싱(Pickle)하여 로딩 오버헤드를 극적으로 감쇄시켰습니다.
*   **사용자 편의성(UX) 및 문맥 인지 극대화**: RAG는 사용자가 명확한 명사 위주로 질문할 때 검색 품질이 향상되지만, 실제 사용자는 "그다음엔?", "거기서 주의할 점은?"처럼 이전 맥락에 의존하는 자연어로 대화합니다. 질문 재구성 모듈을 도입함으로써 RAG 검색 품질의 손해 없이 연속적인 대화형 컨설팅 환경을 제공할 수 있게 되었습니다.

## ✨ 3. 기대 효과 (Expected Benefits)

*   **RAG 답변의 품질 안정성**: 중복 텍스트 조각들이 완전히 정리되어, 항상 겹치지 않고 서로 다른 신뢰할 수 있는 법적 기준 및 가이드 문서 6개가 정확하게 추출되어 Gemini 프롬프트 컨텍스트에 온전히 제공됩니다.
*   **구동 지연 시간 제거**: 프로그램 실행 즉시 디스크에서 사전에 생성된 색인 모델 파일들을 1밀리초 미만 수준으로 초고속으로 메모리에 바인딩하므로 프로그램 시작 렉이 원천적으로 제거됩니다.
*   **유연하고 고도화된 대화 환경 제공**: 사용자가 주어와 주제를 생략해도 이전 대화 맥락을 기반으로 챗봇이 영리하게 문맥을 해석하여 필요한 법령과 시공 주의사항을 물 흐르듯 연속해서 답변해 줍니다.
